# claudeconf Harness Constitution

`constitution-version: 3`

This document states the FIXED, GENERIC invariants every harness generated by the
claudeconf skill must enforce. It names milestones, tiers, and invariants only — it
never hardcodes per-language tool choices (those are researched per project during
skill execution). The `constitution-version` marker above is the machine-readable
conformance profile a manifest's `constitutionVersion` must match; §6 defines when
it bumps and what a bump means for existing harnesses.

---

## 1. Tier Ladder

Every harness is built on a five-tier lifecycle. Each tier has a defined intent and
time budget; checks must be placed at the appropriate tier.

| Tier | Intent | Time budget | Default mode | Default scope |
| --- | --- | --- | --- | --- |
| **edit** | Format + fast auto-fix on the edited file. Silent on success, never blocks. | < 1 s | advisory (autofix) | changed |
| **pre-commit** | Lint, typecheck, secret scan, and unit tests for changed files. | seconds | blocking | changed |
| **pre-push** | Full unit suite, SAST, dependency audit, build. | tens of seconds | blocking | all |
| **gate** | The full battery — every milestone at full scope (see §2). | minutes | blocking | all |
| **CI** | Runs the SAME set as gate on every push to the remote. Authoritative pass/fail signal. | minutes | blocking | all |

**edit is always non-blocking and silent on success.** A failing autofix at the edit
tier must not interrupt the developer's flow; it is logged only when the tool exits
non-zero with output.

**Gate ≡ CI.** The gate tier and CI run the same full-scope milestone set; CI is the
remote, authoritative enforcement of the local gate. Anything that can only fail at
full scope (coverage providers, whole-tree scans, e2e) must therefore fail locally at
the gate first, never for the first time in CI. Every milestone lists both `gate` and
`ci` in its manifest `tiers`.

---

## 2. Milestone Gates

Every generated harness MUST cover all nine milestones. No milestone may be silently
dropped. If a project cannot support a milestone (e.g. no test framework exists yet),
the harness must emit a prominent warning at the relevant tier rather than omitting it.

| Milestone | Default tier(s) | Mode |
| --- | --- | --- |
| format | edit (autofix) + gate + CI (check mode) | advisory |
| lint | pre-commit + gate + CI | blocking |
| typecheck | pre-commit + gate + CI | blocking |
| unit | pre-commit (changed) + pre-push (full) + gate + CI (full + coverage) | blocking |
| e2e | gate + CI | blocking |
| SAST | pre-push + gate + CI | blocking |
| secret-scan | pre-commit (changed) + gate + CI (whole tree) | blocking |
| dependency-audit | pre-push + gate + CI | advisory |
| build | pre-push + gate + CI | blocking |

"Default tier(s)" means the placement a harness uses when no project-specific reason
exists to move a milestone. Every milestone includes gate + CI (the Gate ≡ CI rule,
§1). Tiers may be shifted up (earlier, stricter) but never removed.

---

## 3. Non-Negotiables

These properties are mandatory for every harness, regardless of stack:

1. **Code quality** — format, lint, and typecheck gates are always present and run
   before any commit lands.
2. **Security** — SAST and secret-scan gates are always present. Secret scanning runs
   at pre-commit as a fast first layer; because local hooks are bypassable and do not
   scan history, server-side push protection is the authoritative backstop. The
   default offline rulesets are seed-level — extend them before treating a green check
   as assurance (see `patterns/security.md`).
3. **Autonomous e2e** — end-to-end tests run unattended at gate and CI; they must not
   require manual steps or live external credentials.
4. **GitHub CI on the pipe** — a CI workflow mirrors the gate tier and is the
   authoritative gating signal for pull requests. A harness that lacks a CI workflow
   is incomplete.

## 4. Invariants the Generated Harness Must Obey

### 4.1 Pinned versions

Every tool invoked by a hook must be pinned to an exact version (or a hash) in the
project's lock file or the harness configuration. Floating ranges (e.g. `^1.0`,
`>=2`) are not acceptable for security and SAST tools in particular.

**GitHub Actions are supply-chain inputs and obey this rule too.** Every `uses:` in a
generated workflow must reference a full-length commit SHA, never a mutable tag (`@v4`,
`@main`) — tags can be silently repointed at malicious code (the
`tj-actions/changed-files` compromise). Record the human-readable version in a trailing
comment and let an automated updater (Dependabot/Renovate) bump the SHA.

### 4.2 No network access in the unit tier

Unit tests at the pre-commit and pre-push tiers must not make outbound network
calls. Tests that require network access belong in the e2e milestone (gate + CI).
Hook scripts must not fetch dependencies or tooling at runtime; all dependencies
must be pre-installed.

### 4.3 Deterministic and reproducible

Running the same hook on the same commit twice must produce the same result. Hooks
must not depend on wall-clock time, random seeds, or external service availability.
CI and local runs must use the same pinned tool versions to give identical output.

### 4.4 Idempotent managed-block writes

When the harness writes configuration blocks into existing project files (e.g.
inserting a hook into `.lefthook.yml` or appending to a `.pre-commit-config.yaml`),
it must use managed blocks with sentinel comments so that re-running the harness
is safe: existing managed blocks are updated in place, not duplicated.

### 4.5 Scope guard for pentest engagements

When the harness is generated for a security / pentest context, all network-aware
hooks (e.g. SAST scanners, active scanners, reconnaissance tools) must respect a
`.scope` file at the repository root. The scope file contains one entry per line
(bare hostname or CIDR); comment lines start with `#`. A hook invoking a network
tool must extract the target from the command, check it against the scope entries,
and refuse to run if any target falls outside scope. This prevents accidental
out-of-scope contact during automated checks.

### 4.6 Supply-chain admission (release-age quarantine + script denial)

New third-party code must not reach the project the moment it is published:

1. **Release-age quarantine.** Registry dependencies observe a minimum release age
   on EVERY resolution and update path the ecosystem supports — the package
   manager's install/resolution setting where one exists (e.g. a minimum-release-age
   config) AND the automated updater's cooldown (e.g. Dependabot `cooldown`). One
   without the other leaves a bypass. Security updates MAY bypass the quarantine —
   patching a known vulnerability outweighs the fresh-release risk.
2. **Lifecycle-script denial.** Dependency-supplied install/build scripts are denied
   by default wherever the ecosystem offers auditable allowlisting (e.g. pnpm's
   `strictDepBuilds`/`allowBuilds`); each allowed package is an explicit, recorded
   decision. Where the ecosystem only offers all-or-nothing script blocking that
   would break the project's own workflow (e.g. npm's global `ignore-scripts`
   versus a needed `prepare` hook), record that limitation instead of silently
   accepting default-allow.
3. **Recorded exceptions.** Any quarantine exclusion or script allowance carries a
   short in-config rationale so the exception is reviewable, and an emergency-bypass
   route (e.g. the exclude list) exists for urgent fixes.

The exact settings are researched per ecosystem at generation time; the OUTCOME —
quarantined admission, denied-by-default dependency code execution, recorded
exceptions — is the invariant.

### 4.7 Merge trust root (agent-hardening)

Every harness is assumed to be driven by AI coding agents (`agentUse: true` in the
manifest) unless a human opts out at the generation review gate. The harness is
described as **agent-hardened, never "agent-proof"**, and reports two distinct
states: `configured` (the artifacts exist) and `enforced` (the platform actually
requires them — e.g. branch rules demanding code-owner approval). Configured is
NOT enforced; the harness can scaffold, only humans can enforce.

1. **Independent human control over the control plane.** Every artifact capable of
   weakening the harness requires review by a human owner independent of the
   change author. The protected surface is DERIVED from the generated harness, not
   hardcoded: CI workflows, the hook-runner config, tool configs, manifests and
   lockfiles, scanner exclude/ignore files, provisioning scripts and their
   checksums, the suppression ledger, and the ownership file itself. On GitHub the
   implementation is CODEOWNERS entries plus branch rules (code-owner approval,
   dismiss stale reviews, approval of the latest push, no bypass) — the
   implementation may vary; the outcome may not.
2. **Suppressions are human decisions.** Adding a lint/SAST/secret-scan
   suppression, BROADENING an existing one, or excluding paths/rules at the
   config level requires a recorded human approval at the authoritative merge
   boundary. The mechanism is an approval ledger (see the harness contract): a
   change carrying an unledgered suppression fails; approval is a reviewed change
   to the ledger itself, which lives on the protected surface.
3. **Opting out is protected.** `agentUse: false` may be chosen only at the
   generation human-review gate with a recorded rationale, and disables every
   agent-hardened claim. Flipping the field later is itself a control-plane
   change under rule 1.

Editor-side permission denies (hiding secret files from the agent) are a
default-on, runtime-specific hardening MODULE, not a conformance requirement —
permission rules are defense-in-depth, not isolation; see
`references/agent-hardening.md` for the residual gaps and OS-sandbox guidance.

## 5. Hook-Runner Rule

### Default: lefthook

The default hook runner for every generated harness is **lefthook**. lefthook is
language-agnostic, ships as a single binary, and supports parallel job execution
and glob-based file filtering — properties that make it the right default across
stacks.

### When Claude MAY choose an ecosystem-standard runner instead

Claude MAY substitute the ecosystem-standard runner when **all three** of the
following criteria hold:

1. **Strong ecosystem convention** — the majority of projects in that ecosystem use
   the alternative runner as the de-facto standard (e.g. `pre-commit` in Python).
2. **Existing project commitment** — the project already has a configuration file
   for that runner (e.g. `.pre-commit-config.yaml`) and removing it would cause
   churn without benefit.
3. **Full milestone coverage** — the alternative runner can express every required
   milestone gate with equivalent blocking/advisory semantics; no milestone may be
   demoted or silently dropped to accommodate the runner.

When these criteria hold, Claude documents the choice inline in the generated
harness with a short rationale comment.

In all other cases, use lefthook.

## 6. Versioning & Migration

The `constitution-version` marker at the top of this file names a CONFORMANCE
PROFILE — the bundled constitution + contract requirements a generated harness
satisfies. It is not the claudeconf release number.

- **What bumps it:** changes to normative invariants, required artifacts, the
  manifest schema, the tier model, or acceptance checks — anything that can make a
  previously conforming harness fail. (Version 2 consolidated the tier-default
  tightening plus §4.6 and the provenance requirements; version 3 adds §4.7 —
  the merge trust root, the suppression ledger, `agentUse`, and the versioned
  `hookRunner` schema.)
- **Profiles are archived, not replaced:** the conformance checks for every past
  profile stay bundled with the tooling (keyed by the manifest's recorded
  `constitutionVersion`), so an older harness is validated against ITS profile —
  and offered profile N as "upgrade available" — rather than judged by rules it
  never claimed.
- **What does NOT bump it:** prose clarification, pattern-anchor refreshes, and
  researched tool/version changes — those are upgrades, not profile changes.
- **New generation** always emits the current profile.
- **Existing harnesses** at an older profile remain VALID against their recorded
  profile; a doctor/re-run reports "upgrade available", never "invalid".
- **Migration is explicit:** regenerate the newly required controls, run the full
  self-verify (contract §3), obtain the human-review-gate approval, and only then
  update the manifest's `constitutionVersion`. Never bump the field without the
  controls actually in place.
